# 项目经验记录

> 记录 Vibe Coding 构建 Obsidian KB 自动采集系统过程中的问题和教训，避免重复踩坑。

## 1. B站 WBI 签名：先确认 endpoint 是否需要

**问题**：字幕 endpoint 用了 `x/player/wbi/v2`（需要 WBI 签名），但 PowerShell 始终返回 412。

**根因**：`x/player/v2`（无 WBI）就够了。以为所有 B站 API 都需要 WBI，多此一举。

**教训**：先试无签名的端点，确认需要再加签名。不要假设。

## 2. PowerShell 脚本编码：必须加 UTF-8 BOM

**问题**：脚本里中文字面量写出去全是乱码（"简介" → "绠€浠"）。

**根因**：PowerShell 5.1 读无 BOM 的脚本时，用系统 GBK 解码 UTF-8 字节。

**教训**：所有 PowerShell 脚本文件必须加 `EF BB BF` BOM。PowerShell 7+ 有 `-Encoding UTF8NoBOM` 但 5.1 没有，且 5.1 的 `Out-File -Encoding UTF8` 默认写 BOM。**先确认内存字符串正确，再排查文件编码。**

## 3. SameSite Cookie 是浏览器限制，代码绕不过

**问题**：B站 AI 摘要接口 `conclusion/get` 从浏览器请求总是 -403。

**根因**：SESSDATA 的 `SameSite=Lax` 属性阻止跨子域 (`www` → `api`) Cookie 发送。`GM_xmlhttpRequest` 的 `cookie` 参数理论上能绕过但实际不行。

**尝试的方案**：`document.cookie` → `credentials:include` → `GM_xmlhttpRequest` + cookie 参数 → `unsafeWindow.fetch` → `GM_setValue` 存储 + 注入 → 全部失败。

**教训**：浏览器 Cookie 安全模型是硬限制。如果一定要这个接口，用后端（PowerShell/curl）走。**不要在不确认可行性的方向上花太多时间。** 先验证后端可通，再决定前端是否值得做。

## 4. 字幕内容校验不能省

**问题**：萨特×加缪视频抓到的字幕是"惠普笔记本维修"内容。

**根因**：B站 CDN 的 `auth_key` 偶尔指向错误的内容，或者 `cid` 指向了不同分 P 的字幕。

**解决**：`@clipper` Ingest Step 0 加入字符重叠校验（标题中文 vs 前5条字幕中文，交集 < 10% 则告警）。用户确认后最多重试 3 次。

**教训**：依赖外部 API 的数据一定要有校验层。**B站 API 返回的数据不保证与当前视频一致。**

## 5. 功能改动前先问用户

**问题**：用户说字幕对不上，我直接去掉了字幕功能。

**后果**：用户明确表示不满："请记住不要再犯"。

**教训**：有 bug 修 bug，不要替用户做功能删减的决策。**功能问题和产品需求要分开处理。**

## 6. SPA 导航兼容要三种机制兜底

**问题**：B站是单页应用，切换视频时不刷新页面，脚本 UI 失效。

**解决**：`pushState` 函数劫持 + `popstate` 事件 + 800ms 轮询检测 URL 变化。

**教训**：SPA 导航检测没有银弹。三种机制互备，目前零漏报。

## 7. 文件名：可读性 + 唯一性

**问题**：用户说以标题命名文件更方便查找。

**解决**：`{标题前50字（去非法字符）}_{BVID}.md`。

**教训**：产品设计取舍——可读性放前面（人类友好），唯一性放后面（系统保证）。注意过滤 Windows 文件名非法字符。

## 8. 架构分层：采集与处理解耦

**趋势**：最初想全自动（Tampermonkey 完成一切），后来逐步退化为**采集层（脚本）← 处理层（@clipper 代理）**。

**好处**：两边可独立迭代。采集脚本只拉数据（浏览器限制多也没关系），AI 理解归 @clipper（可以调 LLM 做深度分析和校验）。

## 快速检查清单

新接一个数据源时要确认：

- [ ] 目标 API 是否需要鉴权（Cookie / Token / WBI）？
- [ ] 浏览器端能否发这个请求（CORS / SameSite / 跨域）？
- [ ] 返回的数据是否需要校验一致性？
- [ ] 脚本文件的编码是否正确（Windows 尤其是 UTF-8 BOM）？
- [ ] 文件名是否对人友好且不会冲突？
- [ ] 新功能改动前先问用户是否真的需要？
